# Waxwing Mesh — Social Layer & Gamification Specification

**Version:** 0.1 (Draft)
**Status:** Design phase

This document specifies the optional social and gamification layer of the Waxwing Mesh protocol. All features described here are **opt-in** and have no effect on core mesh operation when disabled. A node with all social features disabled is fully interoperable with nodes that have them enabled.

## 1. Design Intent

The gamification layer serves two distinct purposes:

**Community engagement.** A mesh network is only useful if people participate. Giving users a sense of who they have connected with, how much data has flowed through their node, and where in the world their content has travelled creates a tangible feedback loop that encourages carrying a node, reviewing content promptly, and using the Pass Along action for content outside their direct interests.

**Verifiable encounter records.** For users in journalism, activism, or accountability roles, the ability to cryptographically prove that two specific nodes were physically co-located at a known time — and exchanged data — has real value as a record of provenance. The sync attestation mechanism provides this without requiring any trusted third party or central server.

These two purposes are independent. A user may enable the encounter ledger and companion app visualisations without enabling location features. A user may enable sync attestation without enabling the map. Each feature has its own opt-in flag.

## 2. Encounter Ledger

### 2.1 Overview

Every node maintains a local encounter ledger: a record of every peer node it has synced with, what was exchanged, and aggregate quality signals. The ledger is stored on the SD card and is accessible to the companion app.

### 2.2 Encounter Record Schema

CBOR-encoded, keyed by peer Transport Public Key:

```
{
  "peer_tpk": <bytes>,          // Peer's 32-byte Transport Public Key
  "first_seen": <uint>,         // Unix timestamp of first encounter
  "last_seen": <uint>,          // Unix timestamp of most recent encounter
  "encounter_count": <uint>,    // Total number of sync sessions with this peer
  "bytes_sent": <uint>,         // Total bytes of file content sent to this peer
  "bytes_received": <uint>,     // Total bytes of file content received from this peer
  "files_sent": <uint>,         // Number of distinct files sent
  "files_received": <uint>,     // Number of distinct files received

  // Content quality signals (populated as owner reviews content)
  "rating_sum": <int>,          // Sum of action values for content received from this peer
  "rating_count": <uint>,       // Number of reviewed items from this peer
  // avg_rating = rating_sum / rating_count (computed on read, not stored)

  // Location evidence (populated only if geolocation opt-in is enabled)
  "encounter_locations": [
    {
      "timestamp": <uint>,
      "fingerprint_id": <bytes>  // SHA-256 of WiFi fingerprint; full fingerprint in separate store
    }
  ]
}
```

### 2.3 Encounter Record Updates

The encounter record is updated:

- **On connection:** `last_seen`, `encounter_count`
- **After each file transfer completes:** `bytes_sent` / `bytes_received`, `files_sent` / `files_received`
- **When owner reviews content from this peer:** `rating_sum`, `rating_count`
- **After WiFi fingerprint captured:** `encounter_locations` (if enabled)

Records are never automatically deleted. The owner may manually clear individual records or the entire ledger via the companion app.

## 3. Sync Attestation

### 3.1 Overview

A sync attestation is a cryptographically signed record proving that two specific nodes were physically co-located at a known time and exchanged data. Both nodes hold copies of the complete attestation. Either node can later present it to a third party for verification.

The attestation is produced during the BLE handshake, before data transfer begins. If both nodes have attestation enabled, they exchange signatures as an additional step in the connection sequence.

### 3.2 Attestation Data Structure

```
SyncAttestation {
  "version": 1,
  "session_nonce": <bytes>,       // 16 random bytes generated by the initiating Central; prevents replay
  "timestamp": <uint>,            // Unix timestamp at attestation creation (Central's clock)
  "node_a_tpk": <bytes>,         // Central's Transport Public Key (32 bytes)
  "node_b_tpk": <bytes>,         // Peripheral's Transport Public Key (32 bytes)

  // Signed independently by each party
  "node_a_sig": <bytes>,         // Node A's Ed25519 signature over AttestationPayload_A (64 bytes)
  "node_b_sig": <bytes>,         // Node B's Ed25519 signature over AttestationPayload_B (64 bytes)

  // Location evidence (included only if geolocation opt-in is enabled on each node)
  "node_a_fingerprint": <bytes>, // Node A's serialised WiFi fingerprint (or absent)
  "node_b_fingerprint": <bytes>, // Node B's serialised WiFi fingerprint (or absent)

  // Transfer summary (updated after transfer completes; initially zero)
  "bytes_a_to_b": <uint>,
  "bytes_b_to_a": <uint>,
  "files_a_to_b": <uint>,
  "files_b_to_a": <uint>
}
```

### 3.3 Signature Payloads

Node A signs:
```
AttestationPayload_A = CBOR([
  session_nonce,
  timestamp,
  node_a_tpk,
  node_b_tpk,
  node_a_fingerprint   // empty bytes if geolocation disabled
])
```

Node B signs:
```
AttestationPayload_B = CBOR([
  session_nonce,
  timestamp,
  node_a_tpk,
  node_b_tpk,
  node_b_fingerprint   // empty bytes if geolocation disabled
])
```

Each party signs only their own payload, including their own fingerprint. This means:
- Node A cannot forge Node B's signature (doesn't have B's private key)
- Node B cannot forge Node A's signature
- An observer with both signatures can independently verify each
- If one node has geolocation enabled and the other does not, the attestation is still valid — it simply contains one fingerprint instead of two

### 3.4 Attestation Exchange Protocol

The attestation exchange is performed using the Sync Attestation characteristic (see Section 7) after MTU negotiation and Device Identity exchange, before manifest reading begins:

```
Central (Node A)                           Peripheral (Node B)
  |                                              |
  | generates session_nonce                      |
  | captures WiFi fingerprint (if enabled)       |
  |                                              |
  |── WRITE Attestation: PROPOSE ──────────────>|
  |   {nonce, timestamp, node_a_tpk,            |
  |    node_a_fingerprint (or empty)}            |
  |                                              |
  |                          [B checks: is attestation enabled?]
  |                          [B captures WiFi fingerprint]
  |                          [B computes AttestationPayload_B]
  |                          [B signs with transport private key]
  |                                              |
  |<── NOTIFY Attestation: ACCEPT ──────────────|
  |    {node_b_tpk, node_b_fingerprint,         |
  |     node_b_sig}                             |
  |                                              |
  | [A verifies node_b_sig]                      |
  | [A computes AttestationPayload_A]            |
  | [A signs with transport private key]         |
  | [A assembles complete attestation]           |
  |                                              |
  |── WRITE Attestation: COMPLETE ─────────────>|
  |   {node_a_sig, full attestation record}      |
  |                                              |
  | [Both nodes store attestation locally]       |
  |                                              |
  | [Proceed with manifest exchange...]          |
```

If either node has attestation disabled, it responds to PROPOSE with DECLINE. The handshake continues normally without an attestation record.

### 3.5 Attestation Storage and Access

Attestations are stored on the SD card indexed by `(timestamp, peer_tpk)`. They are accessible to the companion app and can be:

- Viewed in the Sync History screen
- Exported as a signed JSON document for sharing with a third party
- Verified offline using only the two Transport Public Keys and the standard Ed25519 algorithm

### 3.6 Post-Transfer Update

After data transfer completes, the node updates its stored attestation with the final transfer counts (`bytes_a_to_b`, etc.) and signs the update using the same transport key. This produces a `transfer_complete_sig` appended to the attestation, confirming that the transfer summary is authentic.

### 3.7 Third-Party Verification

A third party wishing to verify an attestation needs:
1. The attestation record (exported from either node)
2. The public Transport Keys of both nodes (obtainable from the nodes themselves or published out-of-band)

Verification steps:
1. Check `node_a_sig` is a valid Ed25519 signature over `AttestationPayload_A` using `node_a_tpk`
2. Check `node_b_sig` is a valid Ed25519 signature over `AttestationPayload_B` using `node_b_tpk`
3. Confirm the `session_nonce` matches in both payloads
4. Confirm the `timestamp` is within a plausible range
5. If fingerprints are present, geolocate them to confirm the claimed location

## 4. WiFi Geolocation Fingerprinting

### 4.1 Overview

Nodes without GPS can infer their approximate location by scanning for nearby WiFi access points and recording a fingerprint: a list of SSID and BSSID pairs weighted by RSSI. This fingerprint can later be resolved to geographic coordinates by the companion app using iOS Core Location or a WiFi positioning database.

**This feature is opt-in and disabled by default.** See Section 9 for privacy implications.

### 4.2 Fingerprint Schema

```
WiFiFingerprint {
  "captured_at": <uint>,          // Unix timestamp of scan
  "access_points": [
    {
      "ssid": <string>,           // Network name (may be empty for hidden networks)
      "bssid": <bytes>,           // MAC address (6 bytes)
      "rssi": <int>,              // Signal strength in dBm (negative integer)
      "channel": <uint>           // WiFi channel number
    },
    ...                           // Up to 10 strongest access points, sorted by RSSI descending
  ]
}
```

The fingerprint records up to 10 access points sorted by signal strength. The strongest signals represent the closest physical APs and are the most useful for geolocation.

### 4.3 Fingerprint Capture

Nodes capture a WiFi fingerprint:
- At the start of each BLE sync session (before connection handshake completes)
- When the companion app connects (home location anchor)
- On a configurable periodic schedule (e.g., every 15 minutes) when a sync session is not in progress

On Pico W and CardPuter (ESP32-S3), WiFi scanning is available via the onboard radio. During BLE operation, the node briefly switches the radio to scan mode, captures the AP list, and returns to BLE. Most BLE implementations support this; the scan should complete in under 200ms.

### 4.4 Geolocation Resolution

The companion app resolves fingerprints to coordinates using one of three strategies, in order of preference:

**Strategy 1 — iOS Core Location (recommended)**
When the companion app is displaying the sync map and has location permission, it can request the device's current location (which iOS resolves using GPS, WiFi, and cell triangulation internally). For past fingerprints, the app can use the Apple WiFi geolocation API indirectly: by submitting the BSSID list and using the returned location. This works without GPS and without sending data to a remote server (Core Location processes this on-device).

**Strategy 2 — WiGLE API**
WiGLE (wigle.net) is a community-maintained database of global WiFi access points with coordinates. The companion app may optionally submit BSSID lists to the WiGLE API to resolve coordinates. This requires a network connection and sends BSSID data to a third party. **User must explicitly consent to this strategy.** WiGLE also accepts community contributions — users may optionally upload their discovered APs to improve coverage.

**Strategy 3 — Home anchor**
If the fingerprint matches (by BSSID overlap) the fingerprint captured when the companion app last connected to the node at home, the location is tagged as "home." The user's home coordinates are never stored on the node — only on the companion app — providing a degree of location privacy even if the node is seized.

### 4.5 Fingerprint Privacy

Raw fingerprints (SSID + BSSID) are stored on the node and can reveal movement patterns. To reduce this risk:

- The fingerprint store is separate from the encounter ledger and can be cleared independently
- BSSIDs are hashed (SHA-256) before being stored in attestation records that may be shared externally; the raw fingerprint is kept only in local storage
- The companion app never transmits fingerprints to any server without explicit per-action user consent

## 5. Signed Endorsements and Transitive Trust

### 5.1 Motivation

The reputation model in `PROTOCOL.md §10` operates on direct experience: your node rates content it has received, and those ratings accumulate into a score for each creator. But a node that has never encountered a given creator has no local signal to work from. Transitive trust closes this gap: if you trust a peer you have synced with directly, and that peer has synced with and endorses a third node, you can extend a degree of initial trust to that third node — without ever having met them.

This is the mechanism that makes the community graph useful. Nodes you have never encountered are not unknown strangers; they are members of a network connected to you through people you have met. The graph makes that web visible. The leaderboard makes participation in that web rewarding.

### 5.2 Endorsement Record

A signed endorsement is a compact, cryptographically verifiable statement: *"I have personally synced with this node, and I vouch for it."*

CBOR-encoded:

```
{
  "endorser_tpk": <bytes>,      // Transport Public Key of the vouching node (32 bytes)
  "endorsed_tpk": <bytes>,      // Transport Public Key of the vouched-for node (32 bytes)
  "timestamp": <uint>,          // Unix timestamp when endorsement was created
  "encounter_count": <uint>,    // Number of times endorser has synced with endorsed node
  "endorser_sig": <bytes>       // Ed25519 signature over EndorsementPayload (64 bytes)
}
```

**Endorsement payload (what is signed):**

```
EndorsementPayload = CBOR([
  endorsed_tpk,
  timestamp,
  encounter_count
])
```

Endorsements are created automatically after each successful sync session, if the `social.endorsements` feature flag is enabled. They are updated (new timestamp, incremented encounter count, new signature) each time two nodes sync again.

A node holds one current endorsement per peer it has directly encountered. Old endorsements for the same peer are replaced, not accumulated.

### 5.3 Endorsement Gossip

Endorsements travel alongside reputation scores during the Reputation Exchange characteristic interaction. When two nodes exchange reputation data, each may include a set of its own signed endorsements — acting as introductions for nodes it knows.

**Gossip limits:** A node SHOULD include at most 50 endorsements per reputation exchange, prioritised by:
1. Endorsements for nodes already known to the receiving peer (updates existing trust data)
2. Endorsements for highest-reputation nodes (spread trust for the best actors first)
3. Most recently updated endorsements

**Hop depth:** Endorsements are not re-gossiped indefinitely. Each endorsement carries a `hops_remaining` counter (initial value: 3). Nodes decrement this before forwarding. Endorsements with `hops_remaining = 0` are stored locally but not gossiped further.

This limits the web of trust to a diameter of roughly 3–4 hops, keeping the graph local and the data volume bounded.

### 5.4 Transitive Trust Weight

When computing the effective trust weight for a node in the community graph, hop distance determines a weight multiplier applied to that node's reputation score:

| Source | Trust Weight | Meaning |
|---|---|---|
| Direct encounter (0 hops) | 1.00 | You have personally synced with this node |
| Endorsed by direct peer (1 hop) | 0.75 | Someone you trust has synced with them |
| Endorsed by 1-hop node (2 hops) | 0.50 | A friend of a friend |
| Endorsed by 2-hop node (3 hops) | 0.25 | Outer edge of local community |
| Beyond 3 hops | 0.00 | Not included in local trust graph |

**Effective reputation score** for display purposes:
```
effective_score = raw_reputation_score × trust_weight × recency_factor
```

**Recency factor** decays an endorsement's influence over time, favouring active nodes:
```
days_since_endorsement = (now - endorsement.timestamp) / 86400
recency_factor = max(0.1, 1.0 - (days_since_endorsement / 90))
```

An endorsement loses half its weight after 45 days and 90% of its weight after 81 days. This encourages ongoing participation — nodes that haven't been seen recently gradually fade from the local trust graph.

### 5.5 Trust Path Verification

For any node in the community graph, the companion app can display the **trust path**: the chain of signed endorsements connecting your node to that node. Each link in the chain is independently verifiable using the endorser's public key.

Example path display:
```
Your Node → [direct sync] → Node "WX:A3F7C201"
         → [endorsed] → Node "WX:99DE4F11"
         → [endorsed] → Node "WX:C812A037"   ← currently viewing
```

The companion app verifies each signature in the path when displaying it. A broken or unverifiable link is flagged clearly.

### 5.6 Attestation-Backed Endorsements

A bare endorsement (Section 5.2) is signed only by the endorsing node — a unilateral claim. An **attestation-backed endorsement** is stronger: it references a mutual sync attestation (Section 3) signed by *both* parties, making the vouching claim independently verifiable by anyone with both nodes' public keys.

When `social.attestation` is enabled alongside `social.endorsements`, endorsements automatically include the attestation hash:

```
{
  "endorser_tpk": <bytes>,
  "endorsed_tpk": <bytes>,
  "timestamp": <uint>,
  "encounter_count": <uint>,
  "attest_hash": <bytes>,        // SHA-256 of the most recent full SyncAttestation record
                                 // omitted if social.attestation is disabled
  "endorser_sig": <bytes>        // Signs over AttestationEndorsementPayload (see below)
}
```

When `attest_hash` is present, a different signing payload is used:

```
AttestationEndorsementPayload = CBOR([
  endorsed_tpk,
  timestamp,
  encounter_count,
  attest_hash
])
```

This ensures the attestation hash is cryptographically bound to the endorsement. A verifier can distinguish attestation-backed endorsements from bare endorsements by the presence of `attest_hash`, and must use the appropriate payload when checking the signature.

A receiving node that also holds a copy of the referenced attestation (possible if both have synced with a mutual peer) can verify the claim independently. A receiving node that does not hold the attestation can still accept the endorsement at face value — but the companion app marks attestation-backed endorsements distinctly in the trust path display (solid line vs dashed line in the community graph).

**Critically: geolocation is not required.** The trust value of an attestation-backed endorsement comes from the cryptographic proof that two nodes were physically co-located — not from knowing the geographic coordinates of that encounter. A user with `social.wifi_fingerprint` disabled can still benefit from the full trust propagation mechanism. What they lose is the *location evidence* in the Sync Map; what they keep is the *trust signal* in the community graph and reputation system.

This is by design: the privacy-sensitive feature (location logging) and the trust-building feature (physical proximity proof) are separable. A journalist who needs strong trust propagation for provenance reasons, but must not create a movement record, can enable attestation without enabling fingerprinting.

## 6. Community Graph

### 6.1 Overview

The community graph is a local, private visualisation of your node's known network. It is built entirely from data on your own node and companion app — no external data source, no central server. It represents your slice of the Waxwing Mesh as seen from your vantage point.

The graph has three concentric rings:

- **Ring 0 (centre):** Your own node
- **Ring 1 (inner):** Nodes you have directly synced with
- **Ring 2 (middle):** Nodes endorsed by Ring 1 peers (1-hop trust)
- **Ring 3 (outer):** Nodes endorsed by Ring 2 peers (2-hop trust)

Nodes beyond 3 hops are not shown — the graph stays local and human-scale.

### 6.2 Graph Node Data

Each node in the community graph carries:

```
GraphNode {
  tpk: <bytes>,                   // Transport Public Key (identifier)
  display_name: <string>,         // Node's self-reported name, or key prefix if absent
  hop_distance: 0 | 1 | 2 | 3,   // Distance from your node
  trust_weight: float,            // 1.00 / 0.75 / 0.50 / 0.25
  raw_reputation: int,            // Aggregated reputation score from gossip
  effective_reputation: float,    // raw_reputation × trust_weight × recency_factor
  reputation_tier: string,        // "trusted" | "good" | "neutral" | "poor" | "unknown"
  last_seen: uint | null,         // Unix timestamp; null if never directly encountered
  total_bytes_exchanged: uint,    // 0 if not a direct encounter
  encounter_count: uint,          // 0 if not a direct encounter
  endorsement_path: [tpk],        // Chain of Transport Public Keys from your node to this one
  content_tags: [string]          // Tags most commonly associated with content from this node
}
```

**Reputation tier thresholds** (applied to `effective_reputation`):

| Tier | Range | Colour |
|---|---|---|
| Trusted | > 15 | Green |
| Good | 5 to 15 | Blue |
| Neutral | -5 to 5 | Grey |
| Poor | -15 to -5 | Amber |
| Blocked | < -15 | Red |

### 6.3 Graph Edge Data

Edges connect nodes in the graph. Each edge represents either a direct sync relationship or an endorsement relationship:

```
GraphEdge {
  from_tpk: <bytes>,
  to_tpk: <bytes>,
  edge_type: "direct" | "endorsed",
  weight: float,                  // For direct: proportional to bytes exchanged; for endorsed: trust_weight
  last_active: uint               // Timestamp of most recent sync or endorsement update
}
```

### 6.4 Graph Rendering

The companion app renders the community graph as an interactive force-directed layout:

- **Node size:** proportional to total bytes exchanged (direct peers) or estimated activity (inferred peers)
- **Node colour:** reputation tier colour (see §6.2)
- **Node label:** display name or key prefix, shown on tap
- **Edge thickness:** proportional to edge weight
- **Edge style:** solid line for direct sync relationships; dashed line for endorsement-only relationships
- **Edge colour:** green for trusted path, grey for neutral, amber for poor
- **Animation:** nodes subtly pulse when a new sync has occurred in the last 24 hours

**Interaction:**
- Tap a node → open Node Detail panel (reputation, encounter stats, trust path, content tags)
- Long-press a node → options: view content from this node, block this node, inspect attestations
- Pinch to zoom; drag to pan
- Filter controls: show all hops / direct only / by reputation tier / by content tag

### 6.5 Unattended Nodes in the Graph

Unattended nodes (Relay, Publisher, Archive) appear in the community graph with distinct shapes to make them immediately identifiable:

| Variant | Shape | Meaning |
|---|---|---|
| Relay | Square | Receives and forwards; no human curation |
| Publisher | Arrow / chevron | Pushes content only; a channel or bulletin board |
| Archive | Cylinder | Collects content; does not forward |

Reputation tier colour coding is the same as for human-operated nodes. Tapping an unattended node shows its variant, its configured filter thresholds (if exposed in Device Identity), and its transport reputation. The trust path chain applies to unattended nodes — they can be endorsed by attended peers that have synced with them, but they do not themselves generate endorsements (see PROTOCOL.md §10.7).

Unattended nodes that your node has opted out of syncing with (`sync.unattended_nodes: never`) are shown in the graph as faded/dimmed, so you can see they exist in the network without implying you exchange data with them.

### 6.6 Graph Persistence

The community graph data is rebuilt each time the companion app connects to the node, from the encounter ledger and received endorsements. It is not stored persistently on the node — it is a view computed by the companion app from raw protocol data. This means it is always current and never stale.

## 7. Leaderboard

### 7.1 Overview

The leaderboard is a ranked view of nodes in the community graph, scored across several dimensions. It is designed to make participation tangible and rewarding — showing users not just that the network exists, but who in their local community is most active, most trusted, and most generous.

The leaderboard is entirely local. There is no global ranking, no server, no comparison beyond your own known community. This is intentional: a local leaderboard rewards real-world connections and discourages gaming the system through mass fake node creation (Sybil attack), since fake nodes would need to physically encounter real nodes to appear on anyone's leaderboard.

### 7.2 Leaderboard Categories

#### Most Active — Bytes Shared
Ranked by total bytes exchanged (sent + received combined) across all direct encounters. Rewards nodes that are out in the field moving data through the mesh.

```
score = bytes_sent_to_peer + bytes_received_from_peer
```

#### Most Trusted — Effective Reputation
Ranked by effective reputation score, factoring in hop distance and recency. Rewards consistent positive contributions to the network's content quality.

```
score = effective_reputation (see §5.4)
```

#### Most Connected — Network Footprint
Ranked by the number of unique nodes in the peer's own endorsement network (inferred from gossip). Rewards nodes that travel widely and build bridges between communities.

```
score = unique_nodes_endorsed_by_peer (from received endorsement gossip)
```

#### Most Generous — Pass-Along Rate
Available only if the peer's node has `social.encounter_ledger` enabled and shares aggregate pass-along statistics as part of reputation gossip. Ranked by the percentage of reviewed content passed along or recommended (as opposed to held).

```
score = (recommended_count + passed_along_count) / reviewed_count
```

This category directly rewards the civic behaviour that keeps niche content alive in the network.

#### Explorer — Locations Synced
Available only if both your node and the peer's node have `social.geolocation` enabled. Ranked by the number of distinct geographic locations where sync events have been recorded (using a coarse grid to avoid over-precision).

```
score = distinct_location_cells_visited  // ~1km² grid cells
```

#### Newcomer — Most Recent First Encounter
A rolling 30-day leaderboard of nodes encountered for the first time. Rewards users for going out and finding new nodes — not just repeatedly syncing with the same peers. New encounters are the primary way the network grows.

### 7.3 Leaderboard Display

Each leaderboard shows the top 20 nodes in the selected category. The viewer's own node is always shown even if outside the top 20, with its rank indicated. Each entry shows:

- Rank number
- Node name / key prefix
- Score for the selected category
- Reputation tier colour indicator
- Hop distance badge (direct / 1-hop / 2-hop / 3-hop)
- Last seen timestamp (for direct encounters)

Tapping an entry opens the Node Detail panel from the community graph.

### 7.4 Unattended Nodes and the Leaderboard

Unattended nodes are excluded from all leaderboard categories by default. Competing against automated relay nodes is not meaningful for human participants — a Relay node can exchange many more bytes than a carried node simply by virtue of being plugged in somewhere busy.

Instead, unattended nodes appear in a dedicated **Infrastructure Panel** beneath the leaderboard, ranked by bytes relayed and transport reputation. This gives operators visibility into how their nodes are performing without mixing the metrics with human participation.

The one exception is the **Most Connected** category, where unattended nodes MAY appear if they have accumulated a substantial web of endorsements — since being a well-connected hub in the network is a meaningful contribution regardless of whether it is automated. This is optional and can be toggled off in companion app settings.

### 7.5 Leaderboard as Motivation

The leaderboard is the primary gamification hook for encouraging users to take their node out into the world. The design philosophy:

**Reward effort over luck.** Categories like Most Active and Most Generous reward sustained behaviour, not a single fortuitous encounter.

**Reward diversity.** The Explorer and Most Connected categories reward breadth — meeting new nodes, building bridges — not just depth with existing peers.

**Make the network real.** Seeing a peer's name rise on the leaderboard because you synced with them makes the abstract concept of a mesh network concrete and personal.

**Avoid perverse incentives.** No category rewards raw encounter count alone (which would encourage meaningless brief encounters). Every category rewards something that benefits the network: content quality, generosity, reach, or exploration.

### 7.6 Milestone Badges

In addition to the ranked leaderboard, the companion app awards milestone badges to the owner's node for achievements. Badges are local, never transmitted, and never affect protocol behaviour. They exist purely as a personal record of participation.

| Badge | Trigger |
|---|---|
| First Sync | First successful sync with any peer node |
| Neighbour | 5 unique direct encounters |
| Connector | 25 unique direct encounters |
| Hub | 100 unique direct encounters |
| First Megabyte | 1 MB total bytes exchanged |
| Data Courier | 100 MB total bytes exchanged |
| Generous | Pass-along rate > 60% over 30 days |
| Curator | Recommend rate > 25% over 30 days (quality over quantity) |
| Gatekeeper | Reject rate < 5% over 30 days (discerning but not trigger-happy) |
| Explorer | Synced in 5+ distinct locations |
| Cartographer | Synced in 20+ distinct locations |
| Bridge Builder | Community graph has 3+ disconnected clusters connected through your node |
| Early Adopter | Node created within first 6 months of protocol v1.0 release |
| Relay Operator | Deployed an unattended Relay node that has been active for 30+ days with Good or Trusted reputation |
| Publisher | Deployed an unattended Publisher node that has distributed content to 10+ unique peers |

## 8. Companion App — Social Screens

> *Note: This section describes the full set of social screens. The Community Graph (§6) and Leaderboard (§7) are detailed in those sections above; what follows covers the remaining screens.*

### 8.1 Sync Map

A map view showing approximate locations where sync events occurred. Each sync event is plotted as a dot. Tapping a dot opens the encounter detail for that session.

**Dot colour coding:**
- Blue — sync with a node the owner has encountered before
- Green — first encounter with a new node
- Amber — sync occurred but location is unresolved (fingerprint not yet geolocated)

The map is generated entirely from local data on the companion app. No location data is ever sent to a server.

If location features are disabled, the Sync Map screen shows a message explaining the feature and how to enable it, with a clear description of what data would be stored.

### 8.2 Network Graph

A force-directed graph showing peer nodes as vertices and sync relationships as edges. Edge weight is proportional to total bytes exchanged. Vertices are labelled with the peer node's display name (if set) or the first 8 characters of their Transport Public Key.

Users can tap a vertex to open the Peer Detail screen (Section 8.4).

### 8.3 Sync History

A reverse-chronological list of all sync sessions. Each entry shows:
- Peer node name / key prefix
- Timestamp and duration
- Files and bytes exchanged (sent / received)
- Location (resolved or approximate) if available
- Attestation status (attested / not attested)

Tapping an entry shows full detail and offers the option to export the attestation as a signed JSON document.

### 8.4 Peer Detail

For a specific peer node:
- Node name and Transport Key (abbreviated + copyable)
- Total encounters, first and last seen
- Total bytes sent / received (with a simple bar chart)
- Average content quality: mean action value for content received from this peer, expressed as a label ("Excellent", "Good", "Mixed", "Poor") based on the average
- Encounter timeline
- Option to view attested sync records
- Option to manually trust / block this transport identity

### 8.5 Statistics Dashboard

An aggregate view for the owner's node:

- **Lifetime bytes shared:** total bytes sent to the mesh
- **Lifetime bytes received:** total bytes received from the mesh
- **Unique nodes encountered:** count of distinct Transport Public Keys in the ledger
- **Content pass-along rate:** percentage of reviewed content the owner chose to Pass Along or Recommend (encourages civic participation)
- **Network reach estimate:** sum of unique peers of peers (from reputation gossip) — a rough lower bound on how many nodes the owner's content could reach
- **Milestone badges:** e.g., "First sync", "1 MB shared", "10 unique nodes", "Synced on 3 continents"

Milestone badges are computed locally and never transmitted to any network. They are opt-outable independently of the rest of the social layer.

### 8.6 Attestation Export

From any attested sync record, the owner can export a human-readable + machine-verifiable attestation document (JSON format):

```json
{
  "waxwing_attestation": "1.0",
  "description": "This document certifies that the two Waxwing Mesh nodes identified below were physically co-located and exchanged data at approximately the time and location indicated.",
  "timestamp_utc": "2026-04-01T14:32:11Z",
  "timestamp_unix": 1743515531,
  "node_a": {
    "transport_public_key": "<base64url>",
    "signature": "<base64url>"
  },
  "node_b": {
    "transport_public_key": "<base64url>",
    "signature": "<base64url>"
  },
  "location_evidence": {
    "node_a_wifi_fingerprint": "<base64url of serialised fingerprint>",
    "node_b_wifi_fingerprint": "<base64url of serialised fingerprint>"
  },
  "transfer_summary": {
    "bytes_a_to_b": 204800,
    "bytes_b_to_a": 8192,
    "files_a_to_b": 3,
    "files_b_to_a": 1
  },
  "verification_instructions": "Verify node_a.signature using Ed25519 over the canonical CBOR payload described in Waxwing Mesh Protocol §3.3. Public tools: ..."
}
```

The exported document is self-contained and verifiable by anyone with a standard Ed25519 library. No Waxwing-specific tooling is required to verify it.

## 9. Privacy Considerations and Opt-Out

### 9.1 Feature Flags

Each social feature has an independent opt-in flag stored in node configuration (set via the companion app Device Config characteristic):

| Flag | Default | Controls |
|---|---|---|
| `social.encounter_ledger` | **off** | Encounter counting, bytes tracking, peer list |
| `social.attestation` | **off** | Sync attestation generation and storage |
| `social.endorsements` | **off** | Signed endorsement creation and gossip; required for community graph and leaderboard |
| `social.wifi_fingerprint` | **off** | WiFi AP scanning and fingerprint logging |
| `social.geolocation` | **off** | Fingerprint resolution to coordinates (companion app only) |
| `social.badges` | **off** | Milestone badge computation |
| `sync.unattended_nodes` | `trusted_only` | How this node interacts with unattended peers: `always` / `trusted_only` / `never` / `ask` |

Enabling `social.endorsements` requires `social.encounter_ledger`. Enabling `social.geolocation` requires `social.wifi_fingerprint`. All other flags are independent.

A user running in a hostile environment SHOULD disable all social flags. With all flags off, the node's behaviour is identical to a base protocol node with no additional state recorded.

### 9.2 What Adversaries Can Learn

**If only `social.encounter_ledger` is enabled and node is seized:**
An adversary learns which Transport Public Keys this node has synced with and how much data was exchanged. They do not learn the geographic location of any sync event. They do not learn Content Identity keys (these are never stored on the node unless the user chose stored-mode identity).

**If `social.wifi_fingerprint` is enabled and node is seized:**
An adversary gains a movement history: a sequence of WiFi fingerprints that can be geolocated using the same databases the companion app uses. This is significant. Users in hostile environments MUST keep this flag disabled.

**If `social.attestation` is enabled (without `social.wifi_fingerprint`) and node is seized:**
An adversary gains a list of attested syncs with timestamps and peer Transport Public Keys — but no geographic information. They can prove that two specific nodes synced at a specific time; they cannot determine where. For users who need trust propagation but must not create a movement record (e.g., journalists protecting source locations), this is the recommended configuration: enable attestation, disable fingerprinting.

**If both `social.attestation` and `social.wifi_fingerprint` are enabled and node is seized:**
An adversary gains a list of attested syncs with timestamps, peer Transport Public Keys, and WiFi fingerprints that can be geolocated. This is the highest-risk configuration and MUST NOT be used in hostile environments. The combination provides strong trust propagation and a rich sync map, but at a significant privacy cost.

### 9.3 Data Minimisation

Regardless of opt-in status, the following data is NEVER stored by the social layer:

- Real names, phone numbers, or any PII
- Content Identity keys (creator identities are not part of the social layer)
- The content of files exchanged
- Precise GPS coordinates (only WiFi fingerprints, which require a database to resolve)
- Any data that leaves the device without explicit per-action user consent

### 9.4 Companion App Location Permission

The companion app requests iOS location permission only if `social.geolocation` is enabled by the user. The permission prompt explains: "Waxwing uses location to show you a map of where your node has synced with other devices. Location data stays on your phone and is never uploaded." The app uses "When In Use" location authorization, not "Always" — geolocation resolution happens when the user is actively viewing the sync map.

## 10. GATT Additions

### 10.1 Sync Attestation Characteristic (`CE57580B-494E-4700-8000-00805F9B34FB`)

**Properties:** WRITE, NOTIFY

Used during the attestation exchange protocol (Section 3.4).

Opcodes:
- `0x01` PROPOSE — Central initiates attestation exchange
- `0x02` ACCEPT — Peripheral accepts and provides its signature
- `0x03` DECLINE — Either party declines (feature disabled)
- `0x04` COMPLETE — Central provides its signature and the full attestation record

### 10.2 Endorsement Exchange Characteristic (`CE57580D-494E-4700-8000-00805F9B34FB`)

**Properties:** READ, WRITE

Used to exchange signed endorsements during a sync session. Both nodes read each other's endorsements and incorporate them into their local community graph data.

**READ response:** CBOR array of up to 50 Endorsement Records (§5.2), serialised with an added `hops_remaining` field:

```
{
  "endorser_tpk": <bytes>,
  "endorsed_tpk": <bytes>,
  "timestamp": <uint>,
  "encounter_count": <uint>,
  "hops_remaining": <uint>,     // Decremented before forwarding; 0 = do not gossip further
  "endorser_sig": <bytes>
}
```

The node MUST decrement `hops_remaining` before including an endorsement in a READ response for a peer. Endorsements received with `hops_remaining = 0` are stored locally but MUST NOT be included in subsequent READ responses.

**WRITE:** A node MAY write endorsements it holds to the peer, supplementing what the peer reads. This allows pushing endorsements for nodes that the peer is unlikely to ask about but that may be relevant. Same format as READ response.

### 10.3 Encounter Ledger Characteristic (`CE57580C-494E-4700-8000-00805F9B34FB`)

**Properties:** READ (authenticated — companion only)

Returns the encounter ledger as a CBOR array of Encounter Records, read in chunks using the same offset mechanism as the Manifest Chunk characteristic. Only accessible after companion authentication (see PROTOCOL.md §12.2).
